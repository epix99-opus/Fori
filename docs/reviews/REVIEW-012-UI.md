## VERDICT: CONDITIONAL_PASS

## FINDINGS

F-001. 六大 PRD 模块中，模块六“Agent 原生智能化技术底座体系”缺少明确的用户可见页面或运营入口。`docs/UI_DESIGN.md` 覆盖了楼盘字典、房源客源匹配、信用认证/交易公证、自媒体营销、房价评估五类业务模块，但模块六在 PRD 中包含 Agent 编排、业务智能体、版本同步、高并发运维等能力，UI 文档未说明这些能力是否仅为后台技术底座，或是否需要管理员/运营可视化入口。若模块六不需要前端页面，应在 UI 文档中显式标注“无独立用户页面，通过各业务页后台能力体现”。来源：`docs/PRD.md` 模块六；`docs/UI_DESIGN.md` 页面 01-21。

F-002. PRD 中“公证合规第三方机构”是独立服务对象，但 UI 设计只覆盖买卖双方/经纪人视角的交易流程页与公证存证页，缺少公证机构接收核验任务、查看资料包、回传结果、导出存证/纠纷材料的机构端页面或入口。若该角色只通过 API 接入，也需要在信息架构和权限矩阵中明确“不提供人工 UI，仅提供 API/后台接口”。来源：`docs/PRD.md` 2.5、3.2；`docs/UI_DESIGN.md` 页面 13、14。

F-003. 21 个核心页面并非每页都满足“页面名称/路径/所属模块、页面目的、用户角色、页面布局描述、交互流程、数据展示内容、交互元素、空状态/加载/错误状态处理、页面间跳转关系”的完整字段要求。页面 12-14、16-21 多数只有布局描述，缺少独立的“交互流程”“状态处理/空加载错误”“页面间跳转关系”小节；页面 12 信用档案、页面 14 公证存证、页面 17 经纪人工作台、页面 18 门店管理、页面 20 个人中心、页面 21 设置页尤其明显。当前文档可指导粗略原型，但不能稳定支持逐页高保真交互原型绘制。

F-004. 认证与登录链路在页面层级树中存在，但未纳入 21 个核心页面详细设计。`/auth/login` 和 `/auth/kyc` 是游客态、实名认证、经纪人认证、发布房源、交易流程的前置关口，但核心页面章节只详细展开了 `/onboarding` 与 `/profile/agent-cert`，没有登录/注册、验证码、第三方登录、实名认证失败/重试、身份材料补充等原型级交互。来源：`docs/UI_DESIGN.md` 1.2、页面 01、页面 11。

F-005. PRD 模块二要求“房源客源甄别流存与智能精准匹配体系”，但 UI 详细页面只覆盖发布房源、发布买家需求、智能匹配推荐和经纪人工作台概览；页面层级树列出的 `/workspace/agent/buyers`、`/workspace/agent/listings`、`/workspace/agent/stats` 未进入 21 个核心页面详细设计。客源库沉淀、房源管理、跟进记录、历史流存、失效/转化状态等关键操作缺少可画原型的页面规格。来源：`docs/PRD.md` 2.1-2.4；`docs/UI_DESIGN.md` 1.2、页面 08-10、17。

F-006. 独立房价评估入口 `/price` 在信息架构中存在，但详细页面 07 主要描述 `/listing/:listing_id/price` 或 `/price/result`，没有定义从首页独立入口进入时的搜索小区、选择房源、手动录入评估参数的页面布局和错误状态。虽然 Mermaid 流程提到“首页功能入口 → 搜索目标小区 → 手动输入楼层/面积/朝向/装修”，但缺少对应页面级原型描述。来源：`docs/UI_DESIGN.md` 1.2、页面 07、4.5。

F-007. 关键流程 Mermaid 覆盖了买家购房、卖房发布、经纪人入驻、楼盘字典共建、房价评估、交易+公证六条主流程，满足数量要求；但流程节点与页面规格之间缺少映射表。比如“确认购买意向”“经纪人创建交易记录”“补充材料”“合同签署”“资金监管账户进度”等节点没有明确落在哪个页面、由哪个角色触发、失败后返回哪个状态。来源：`docs/UI_DESIGN.md` 4.1-4.6。

F-008. 移动端特有交互覆盖了手势、转场动画、离线缓存、推送通知，维度完整；但存在两处需要修正的可实现性问题：全局手势表把“左滑（屏幕边缘 20px 向右滑）”命名与方向描述冲突，应统一为“右滑返回”；地图手势中 Pinch-out/Pinch-in 的缩放描述与常见移动端语义相反，需校正以避免原型和开发实现不一致。来源：`docs/UI_DESIGN.md` 5.1。

F-009. 全局设计规范具备色彩色值、字体系统、间距系统、组件库、图标系统，满足基础完整性；但组件库仍缺少表单 Stepper、Tab、Segmented Control、地图房源气泡、图表容器、上传组件、时间选择器、权限受限提示、空状态组件、错误页组件等高频业务组件规范。由于多个核心页面依赖这些组件，当前组件库不足以直接沉淀为设计系统或前端组件清单。来源：`docs/UI_DESIGN.md` 2.2-2.6、页面 04-18。

F-010. 页面状态策略过度依赖全局骨架屏、Toast 和无限滚动规则，逐页缺少业务错误和恢复路径。典型缺口包括：房源详情下架/不存在、搜索无结果与服务失败、楼盘字典无权限编辑、价格评估计算失败、交易流程材料被驳回、存证证书验签失败、推广平台授权过期、门店无成员/无报表数据。缺少这些状态会影响原型评审和异常路径验收。

## REQUIRED_CHANGES

1. 在 `docs/UI_DESIGN.md` 增加“PRD 模块到页面覆盖矩阵”，逐项列出六大模块、PRD 功能点、对应页面/入口、覆盖状态；对模块六和公证机构端明确“无独立 UI / 后台管理 / API 接入”的产品决策。

2. 为 21 个核心页面统一补齐页面规格模板：页面名称、路径、所属模块、页面目的、适用角色、布局、数据展示、交互元素、交互流程、空状态、加载状态、错误状态、权限状态、跳转关系。已有内容可保留，但必须按同一结构补齐缺口。

3. 补充登录/注册页、实名认证页、独立价格评估入口页、客源管理页、房源管理页、成交统计页的原型级设计；若不纳入 21 个核心页面，应增加“非核心但必需页面”章节。

4. 为六条 Mermaid 关键流程增加“流程节点 → 页面/组件 → 角色 → 失败/回退路径”映射表，确保流程图可直接驱动原型连线。

5. 扩展组件库，至少补齐 Stepper、Tab、筛选 Bottom Sheet、上传/OCR、地图气泡、图表卡、交易时间线、证书卡、权限提示、空状态、错误状态、离线队列提示等业务组件。

6. 修正移动端手势方向描述，并补充手势冲突规则，例如横向图片轮播与边缘返回、Bottom Sheet 下滑关闭与列表滚动、地图拖动与页面滚动的优先级。

## SUGGESTIONS

1. 建议把 21 个核心页面拆成“买家端、卖家端、经纪人端、门店端、合规/公证端、全局基础页”六组，便于后续设计评审按角色走查。

2. 建议增加低保真线框图索引或 Figma 页面命名规范，例如 `P03 Listing Detail / P13 Transaction Timeline`，避免设计与开发路由脱节。

3. 建议在每个页面增加“最小可用数据契约”，列出页面依赖的 API 数据字段、权限字段和本地缓存字段，方便前端原型与后端接口对齐。

4. 建议把异常状态作为验收清单独立维护，尤其是房源下架、交易中止、公证拒绝、营销分发失败、离线提交冲突等高风险业务场景。
