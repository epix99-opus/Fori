# Fori 用户视角 UX 评审报告

**文档路径**：`docs/reviews/REVIEW-UX-USER-PERSPECTIVE.md`  
**评审角色**：Codex（用户体验评审专家）  
**评审日期**：2026-07-01  
**评审范围**：`docs/UI_DESIGN.md`、`docs/PRD.md`  
**评审方法**：按买家、卖家、经纪人、门店管理、公证机构五类角色，从端到端流程、信息充分性、关键决策点、异常路径和下一步 CTA 进行走查。

---

## 总体结论

Fori 的 PRD 与 UI 设计已经覆盖主要业务链路，页面、权限、状态、失败回退路径也有较完整映射。主要 UX 风险不在“功能是否存在”，而在高信任交易场景下，用户是否能明确知道下一步由谁发起、需要等多久、当前风险是什么、费用和法律后果是什么。尤其是交易发起、公证核验、费用承担、房源管理、客源管理和机构端 API 操作面，仍需要更明确的用户任务视角设计。

## 买家视角

### UX-01：从预约看房到确认购买意向的交易发起责任不清

- **角色视角**：买家
- **流程步骤**：首页 → 搜索 → 房源详情 → 在线沟通 / 预约看房 → 价格评估 → 确认购买意向 → 交易流程
- **体验问题描述**：买家全流程图中有“确认购买意向”，交易公证流程中却由“经纪人创建交易记录”发起。房源详情页的底部 CTA 是电话、在线沟通、预约看房，没有“发起购买意向 / 出价 / 邀请创建交易”的明确动作。用户在看完价格评估并决定推进后，不知道是自己发起、经纪人发起，还是必须先在线沟通。
- **严重程度**：P1 重要
- **改进建议**：在房源详情和价格评估页增加“发起购买意向”CTA，提交后进入意向确认草稿，由买家填写出价、付款方式、看房记录和有效期；系统通知卖家与经纪人确认，确认后自动生成交易记录。交易流程页顶部明确“本交易由谁发起、当前等待谁确认”。

### UX-02：价格评估结论不足以支撑买家出价决策

- **角色视角**：买家
- **流程步骤**：房源详情 → 价格评估 → 查看修正项 → 决定继续谈价或确认意向
- **体验问题描述**：价格评估页展示估价区间、瀑布图和走势图，但买家真正需要的是“我该出多少钱、议价空间多大、风险在哪里”。当前设计给出“偏高 / 偏低 / 合理”结论，但未明确建议出价区间、置信度、样本数量不足时的可信度提示，以及税费 / 资金成本对总购房成本的影响。
- **严重程度**：P1 重要
- **改进建议**：为买家视角增加“建议出价区间”“可谈价空间”“总成本估算”“样本可信度”四个信息块。对成交样本不足、官方数据缺失、同层级样本偏少的情况用醒目提示说明“仅作参考，不建议直接出价”。

### UX-03：搜索无结果后的买家留存路径偏弱

- **角色视角**：买家
- **流程步骤**：首页 → 搜索筛选 → 无结果 / 结果过少 → 买家需求发布 → 等待匹配
- **体验问题描述**：搜索页有“放宽筛选”和热门小区兜底，买家需求发布页也存在，但两者衔接不够强。用户筛选多次无结果时，系统没有自动把筛选条件带入“发布购房需求”，也没有说明发布需求后多久会有响应、谁会联系、能否撤回。
- **严重程度**：P2 建议
- **改进建议**：搜索无结果页增加“用当前条件发布购房需求”主 CTA，自动带入区域、户型、预算和特殊要求；成功页展示预计匹配时间、匹配规则、可撤回入口和隐私说明。

### UX-04：交易流程中买家需要承担的费用与法律承诺展示太晚

- **角色视角**：买家
- **流程步骤**：确认购买意向 → 实名认证 / 购房资格核验 → 公证前置核验 → 合同签署 → 资金监管
- **体验问题描述**：PRD 对买卖双方默认各承担 0.5% 服务费、公证存证 5% 从平台服务费列支、资金监管合规有明确口径，但 UI 交易流程主要展示时间线和税费明细。买家在确认意向前需要看到服务费、税费、资金监管、存证不可删除、违约后果等关键承诺，否则容易在合同节点才发现成本和义务。
- **严重程度**：P1 重要
- **改进建议**：在“确认购买意向”前增加交易预确认页，展示总价、服务费承担方式、预计税费、资金监管路径、公证存证告知书、合同关键条款摘要和撤回规则；要求用户逐项确认后再进入正式交易。

## 卖家视角

### UX-05：发布房源 5 步表单合理，但对低数字能力房东仍偏重

- **角色视角**：卖家
- **流程步骤**：实名认证 → 点击发布房源 → 选择楼盘 → 填写房屋信息 → 上传图片 → 填写挂牌价 → 上传房产证 → 提交核验
- **体验问题描述**：发布房源页需要 5 步，包含楼盘字典、房屋参数、图片、价格、产权材料。对中老年房东或首次卖房用户而言，字段较多且术语偏专业，如容积率、产权年限、税费承担、共有产权声明。当前有草稿保存和 OCR，但缺少“快速发布 / 经纪人协助发布”的分流。
- **严重程度**：P1 重要
- **改进建议**：发布入口先分流为“自己发布”和“请经纪人协助发布”。自己发布默认只收最小必填项，复杂字段从楼盘字典自动带出；经纪人协助模式允许房东授权指定经纪人补全非敏感字段，敏感证件仍由房东本人上传确认。

### UX-06：挂牌价决策缺少对卖家收益的闭环解释

- **角色视角**：卖家
- **流程步骤**：发布房源 Step 4 → 查看价格参考 → 调整挂牌价 → 提交核验 / 上架 → 后续调价
- **体验问题描述**：价格参考可以提示片区均价和估价偏差，但卖家决策不仅关心“价格合理”，还关心预计成交周期、降价概率、扣除服务费和税费后的净到手金额。当前发布页的价格参考没有和卖家收益、成交速度、推广策略形成一个决策面板。
- **严重程度**：P1 重要
- **改进建议**：在挂牌价步骤增加“卖家净收益测算”：预计成交价区间、预计成交周期、服务费承担方式、税费估算、净到手金额。挂牌价偏高时提示“可能减少匹配推荐 / 延长成交周期”，并提供一键生成三档定价策略：快售、均衡、试探。

### UX-07：房源提交后等待核验的过程反馈不足

- **角色视角**：卖家
- **流程步骤**：提交房源 → 系统核验队列 24h → 通过 / 待确认 / 未通过 → 补材料 / 修改
- **体验问题描述**：PRD 和 UI 都说明核验预计 24 小时，但提交后的状态页只描述“进入核验队列”。卖家会关心当前卡在哪个环节、是否需要人工、公证机构是否参与、如果超过时限怎么办。缺少核验进度条和超时承诺会造成焦虑。
- **严重程度**：P2 建议
- **改进建议**：提交成功页展示核验时间线：材料上传、OCR、产权比对、公证 / 平台复核、上架。每一步给出预计完成时间、失败原因和补救入口；超过承诺时间自动创建客服工单并通知卖家。

### UX-08：推广素材生成与房源管理之间的闭环不够显性

- **角色视角**：卖家
- **流程步骤**：房源上架 → 生成推广素材 → 分发平台 → 查看推广数据 → 调整图片 / 文案 / 价格
- **体验问题描述**：推广素材生成页和推广管理页功能完整，但卖家视角缺少“推广效果不好时下一步怎么做”的闭环。曝光、互动和线索数据如果只展示统计，卖家难以判断应该补图、调价、换文案还是停投。
- **严重程度**：P2 建议
- **改进建议**：推广管理页增加“诊断建议”：曝光低建议更换封面或渠道，点击低建议调整标题，咨询少建议调价或补充卖点。房源管理页同步展示推广效果摘要，并提供“重新生成素材 / 调价 / 刷新房源”快捷操作。

## 经纪人视角

### UX-09：经纪人入驻认证权益清晰，但首日工作路径不足

- **角色视角**：经纪人
- **流程步骤**：经纪人入驻认证 → L1 / L2 权益解锁 → 工作台 → 开始维护楼盘 / 接收客源
- **体验问题描述**：经纪人认证页有层级、权益表和 Stepper，但认证完成后，工作台只展示概览卡和快捷入口。新入驻经纪人不知道第一天该做什么：先维护哪个小区、补多少字段能进 Top3、多久能收到客源、如何开启推送。
- **严重程度**：P1 重要
- **改进建议**：L2 认证通过后进入“新经纪人 3 步启动任务”：选择深耕片区、维护 3 个小区关键字段、开启客源推送。工作台空状态用任务卡展示“完成后预计获得哪些优先权益”。

### UX-10：智能匹配卡片缺少经纪人的收益和跟进成本判断

- **角色视角**：经纪人
- **流程步骤**：工作台 → 智能匹配推荐 → 查看 P1 / P2 客源 → 接受跟进 → 解锁联系方式 → 沟通
- **体验问题描述**：匹配卡显示区域、户型、预算、匹配度、意向强度和倒计时，但经纪人需要快速判断“这个客源值不值得抢”。当前缺少成交概率、预计佣金、竞争人数、客户是否已看过类似房源、需要响应的承诺标准。
- **严重程度**：P1 重要
- **改进建议**：客源卡增加成交概率、预计佣金区间、可推荐房源数、已推送经纪人数、最近活跃行为。接受前弹窗说明跟进 SLA，例如多久内联系、放弃规则、超时对信用分影响。

### UX-11：楼盘字典共建价值明确，但贡献排名和权益兑现不够可操作

- **角色视角**：经纪人
- **流程步骤**：工作台 → 楼盘字典搜索 → 编辑小区 / 楼栋 / 单套 → 提交核验 → 获得积分 / Top3 优先匹配
- **体验问题描述**：PRD 明确 Top3 维护经纪人获得优先匹配权，UI 有版本提示和维护积分，但编辑页没有说明当前排名、距离 Top3 还差哪些字段、哪些字段更影响匹配权益。经纪人可能感觉在“填资料”，而非在“换取客源”。
- **严重程度**：P2 建议
- **改进建议**：在楼盘字典详情和编辑页展示“贡献权益面板”：当前贡献排名、Top3 差距、可补字段清单、字段权重、预计解锁客源权益。提交成功后明确“本次贡献提升了什么权益”。

### UX-12：客源管理、房源管理、成交统计虽有必需页面，但工作台主流程容易过度分散

- **角色视角**：经纪人
- **流程步骤**：工作台 → 客源管理 / 房源管理 / 推广管理 / 成交统计 / 交易流程
- **体验问题描述**：经纪人日常工作高频切换客源、房源、推广、交易和统计。当前工作台是聚合入口，但缺少“今日优先级队列”，不同模块的待办可能分散在消息中心、匹配页、交易页和房源管理页，导致经纪人漏处理 P1 客源或交易节点。
- **严重程度**：P1 重要
- **改进建议**：工作台首屏增加“今日必须处理”统一队列，按超时风险排序：P1 客源倒计时、交易待签署、房源核验驳回、推广授权失败、楼盘审核未通过。每条待办直接跳到对应处理页，并支持完成后返回队列。

## 门店管理视角

### UX-13：门店成员管理缺少邀请后的审核、入职和离职闭环

- **角色视角**：门店管理
- **流程步骤**：门店管理 → 成员管理 → 邀请经纪人 → 设置权限 → 调整权限 / 踢出门店
- **体验问题描述**：门店管理页支持成员列表、邀请码 / 手机号邀请、调整权限和踢出门店，但缺少邀请状态、经纪人接受确认、资质未通过处理、离职后的客源 / 房源 / 交易归属转移。门店管理员会担心误踢或人员变动导致业务资产丢失。
- **严重程度**：P1 重要
- **改进建议**：成员管理增加完整生命周期状态：已邀请、待接受、认证中、在职、停用、已离职。离职 / 踢出前必须展示资产交接清单：客源、房源、楼盘维护权限、未完成交易，并指定接收人后才能生效。

### UX-14：门店报表能看结果，但缺少管理动作建议

- **角色视角**：门店管理
- **流程步骤**：门店管理 → 业绩报表 → 查看成交 / 佣金 / 排行 → 调整团队策略
- **体验问题描述**：业绩报表展示成交套数、佣金、成员排行、楼盘维护进度，但管理员的目标是发现问题并采取行动。当前报表偏结果展示，缺少异常提示和下一步操作，例如某经纪人客源响应超时多、某片区维护不足、某房源推广转化差。
- **严重程度**：P2 建议
- **改进建议**：报表页增加“管理洞察”：响应超时榜、维护空白片区、成交漏斗掉点、低转化房源、信用风险成员。每个洞察提供动作入口：分配楼盘、提醒经纪人、调整权限、导出复盘。

### UX-15：门店与个人经纪人的收益关系说明不足

- **角色视角**：门店管理
- **流程步骤**：门店管理 → 成交统计 / 佣金汇总 → 成员业绩排行 → 收益结算
- **体验问题描述**：PRD 说明门店参与成交时，经纪人 80% 份额先结算至服务经纪人所属结算主体，门店与经纪人的内部分账按入驻协议执行。但 UI 门店管理页没有呈现结算主体、内部分账规则、待结算 / 已结算状态，容易在成交后产生财务争议。
- **严重程度**：P1 重要
- **改进建议**：门店报表和收益结算入口展示每笔成交的结算主体、平台服务费、经纪人应得、门店内部分账规则、税务主体和结算状态。成员加入门店时必须确认分账协议，后续变更需留痕。

## 公证机构视角

### UX-16：公证机构端被定义为 API-only，但任务处理体验没有自助校验面

- **角色视角**：公证机构
- **流程步骤**：公证任务接收 → 核验资料包 → 回传结果 → 存证证书签发
- **体验问题描述**：UI 设计明确公证机构端不提供独立人工 UI，机构通过 API 接收任务、回传结果和导出材料。这个决策可以降低主产品复杂度，但从机构接入和日常运维视角看，缺少 API 任务状态、失败重试、资料包完整性、回传验签结果的自助检查界面。完全依赖接口和平台运维后台，机构处理问题时可见性不足。
- **严重程度**：P1 重要
- **改进建议**：即便不做完整机构业务端，也应提供“机构接入控制台 / API Console”：任务列表只读、任务详情脱敏预览、资料包完整性校验、回调日志、签名验证、失败重试、SLA 统计。该控制台可作为 PC 响应式后台，不进入移动端主导航。

### UX-17：PENDING 补材料流程对公证机构和交易当事人的责任边界不够清晰

- **角色视角**：公证机构
- **流程步骤**：公证核验 → 返回 PENDING → 双方补材料 → 再次提交核验
- **体验问题描述**：PRD 定义公证机构可返回 `PENDING` 并附补充清单，UI 交易流程页显示补材料入口。但没有规定补充清单的标准字段、由谁解释、谁审核补充材料是否满足要求、再次提交后 SLA 是否重新计算。机构和平台客服可能重复沟通，买卖双方也容易不知道应该补什么。
- **严重程度**：P1 重要
- **改进建议**：标准化补材料清单 schema：材料名称、责任方、示例、截止时间、是否必须、驳回原因、重新提交次数。交易页按责任方拆分补材料任务，公证机构只需回传结构化字段，平台负责向用户解释和采集。

### UX-18：争议处理路径对公证机构而言缺少材料包版本和导出标准

- **角色视角**：公证机构
- **流程步骤**：争议申请 → 平台生成纠纷支撑材料包 → 公证机构调取存证 → 出具纠纷认定意见书
- **体验问题描述**：PRD 描述争议时平台调取全部存证记录并提交公证机构，但 UI 只在存证页提供“申请纠纷”。缺少机构侧材料包版本、导出格式、证据目录、材料补正、意见书回传和当事人可见范围。争议处理属于高风险路径，信息组织必须更严格。
- **严重程度**：P1 重要
- **改进建议**：定义“纠纷材料包”交互和数据标准：证据目录、材料版本号、生成时间、哈希校验、当事人提交材料、平台审计日志、导出 PDF / ZIP、机构意见书编号和回传状态。用户端存证页展示争议材料包状态，机构控制台展示处理队列。

### UX-19：存证证书用户可读性较好，但验签失败后的信任修复路径不足

- **角色视角**：公证机构
- **流程步骤**：存证完成 → 用户查看证书 → 复制哈希 / 下载 PDF / 分享 → 验签失败或下载失败
- **体验问题描述**：存证页展示编号、哈希、时间戳、数字签名和下载操作，基础信息充分。但当验签失败、签名异常、下载失败时，UI 只描述重试和客服，没有说明是否影响法律效力、是否需要公证机构复核、复核期间交易证书如何标记。
- **严重程度**：P2 建议
- **改进建议**：增加“证书异常处理”状态：验签中、验签失败待复核、机构复核中、已恢复、证书撤销。异常时在用户端明确“交易状态不受影响 / 暂停使用证书 / 等待机构复核”的差异，并自动创建公证机构复核任务。

## 优先级建议

1. **P1 优先修复**：交易发起责任、买卖双方费用与承诺确认、卖家发布分流、经纪人今日任务队列、门店成员资产交接、公证机构 API Console、补材料结构化。
2. **P2 可随原型细化推进**：搜索无结果转需求、核验等待反馈、推广诊断、楼盘贡献权益面板、门店管理洞察、证书异常状态。
3. **建议新增验收走查**：按五类角色分别录制端到端原型路径，要求每个关键节点都能回答“我现在在哪、我为什么要做、下一步谁做、失败去哪、要等多久、费用/法律后果是什么”。
