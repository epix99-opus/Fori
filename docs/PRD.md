## 1. 产品概述

### 1.1 核心愿景

打破传统房地产中介行业**信息垄断、房源虚假、定价混乱、交易成本高、信用缺失、收益分配不均、交易争议频发**七大核心痛点，依托 Agent 原生技术架构，打造**真实、合规、信用、低成本、多方共赢、智能自动化**的新一代房产交易中介生态平台。

### 1.2 战略定位

Fori 不是传统意义上的房源信息聚合平台，也不是单纯的中介撮合工具。Fori 的战略定位是：

> **以数据共建为基础，以信用公证为保障，以科学定价为锚点，以智能营销为引擎，以四方共赢为动力，构建中国下一代房产交易中介生态操作系统。**

Fori 的核心竞争优势不在于垄断信息，而在于**开放信息、共建信息、核验信息、共享信息**，用透明代替垄断，用生态代替零和博弈。

### 1.3 核心差异化优势

| 维度 | 传统中介 / 现有平台 | Fori 平台 |
|------|-------------------|-----------|
| 房源数据 | 各家私有、信息孤岛、重复虚假 | 全民共建、透明共享、协同维护 |
| 价格体系 | 经验主义、宏观城市分级、模糊混乱 | 独创在地分层、微观精准、可视可拆解 |
| 合规体系 | 自建合规体系、成本高、争议多 | 完全对接国家级官方体系，极简合规 |
| 收益分配 | 平台高抽成、经纪人收益被压缩 | 低抽成、收益向一线经纪人倾斜 |
| 交易保障 | 口头协议、纸质存证、事后争议难维权 | 全程第三方公证存证、数字化维权 |
| 营销能力 | 依靠人工发帖、效率低、覆盖窄 | 智能素材生成、一键全渠道分发 |
| 技术架构 | 传统单体或微服务架构 | Agent 原生、智能自动化、高并发稳定 |

### 1.4 服务对象

- **个人房东**：持有住宅产权的卖方，需要高效出售或出租房产
- **购房买家**：有购房意向的个人，需要真实房源信息和安全交易保障
- **入驻经纪人**：在 Fori 平台注册的个人房产经纪从业者
- **中介门店**：在 Fori 平台入驻的线下房产中介机构
- **公证合规第三方机构**：提供官方公证、存证、合规核验服务的机构

## 2. 用户角色定义

### 2.1 个人房东（卖方）

**角色定义**：持有一套或多套住宅产权，希望通过 Fori 平台出售或出租房产的自然人。

**核心画像**：
- 年龄分布：30～70 岁为主
- 技术能力：中等，能使用手机浏览器或安装到桌面的 PWA 操作基本功能
- 核心诉求：快速、安全、无套路地卖出房产，减少被中介"吃差价"的风险
- 痛点：不知道房子真实市场价、担心中介乱承诺、不放心签署复杂合同、交易周期漫长

**核心需求**：
1. 上传并核验房源真实性，获得平台合规背书
2. 获取独立、科学的在地分层定价参考
3. 智能推广素材一键分发，主动触达潜在买家
4. 全流程透明、可追溯、第三方公证保障权益
5. 佣金透明，无隐性收费

**使用场景**：
- 场景 A：房东自主发布房源 → 选择接受经纪人服务 → 配合核验房产证 → 等待客源匹配
- 场景 B：房东上传房源 → 查看平台给出的定价图谱 → 自主调整挂牌价 → 发布营销素材
- 场景 C：房东收到意向买家 → 确认公证机构介入 → 完成网签过户 → 查看交易存证

**权限范围**：
- 可访问：房源发布、定价参考、营销推广、交易进度、存证记录
- 不可访问：其他经纪人的客源信息、其他卖家的私有通信记录

---

### 2.2 购房买家

**角色定义**：有明确购房意向、希望在 Fori 平台找到真实匹配房源并完成安全交易的自然人。

**核心画像**：
- 年龄分布：25～55 岁为主，互联网使用熟练
- 技术能力：中高，习惯线上信息获取
- 核心诉求：买到货真价实、产权清晰、价格合理的房子，不被虚假房源浪费时间，不被套路伤害权益
- 痛点：看了大量虚假房源、定价参考不准、中介随意加价、合同条款不透明

**核心需求**：
1. 快速筛选真实有效的房源，过滤虚假信息
2. 获取独立的在地分层定价参考，避免多付款
3. 清晰了解房产产权状态、税费情况
4. 全程有第三方公证保障，签约无后顾之忧
5. 交易结果可追溯、可维权

**使用场景**：
- 场景 A：买家设置购房需求 → 平台智能匹配精准房源 → 联系经纪人看房 → 在平台完成意向确认
- 场景 B：买家查看目标小区 → 查阅在地分层定价图谱 → 判断挂牌价是否合理 → 决策出价
- 场景 C：买家签订购房意向 → 公证机构介入核验 → 完成官方网签 → 查看公证存证

**权限范围**：
- 可访问：公开房源信息、定价图谱、经纪人联系方式、交易存证（己方）
- 不可访问：其他买家的私有需求信息、经纪人的完整客源库

---

### 2.3 入驻经纪人

**角色定义**：在 Fori 平台完成资质认证、独立执业的个人房产经纪从业者，是平台最核心的服务供给方。

**核心画像**：
- 从业年限：1～20 年不等
- 技术能力：中等偏高，依赖手机端高频操作
- 核心诉求：多获客、多成交、多赚钱，减少无效工作，最大化自己的劳动收益
- 痛点：大平台抽成过高、优质客源被截流、楼盘数据不共享导致重复劳动、营销成本高

**核心需求**：
1. 维护深耕楼盘数据，获得优先客源匹配权益
2. 管理个人客源库，沉淀长效资产
3. 一键生成专业营销素材，降低推广成本
4. 平台低抽成，收益最大化
5. 信用档案累积，建立个人品牌

**使用场景**：
- 场景 A：经纪人维护目标小区楼盘字典 → 获得该楼盘优先匹配客源 → 跟进成交 → 收取佣金
- 场景 B：经纪人录入新客源需求 → 系统自动匹配全库房源 → 推送匹配结果 → 安排看房
- 场景 C：经纪人选择房源 → 一键生成视频、图文、文案 → 分发到抖音、微信、小红书等平台
- 场景 D：经纪人协助客户进入交易流程 → 公证机构介入 → 完成合规交易 → 获取佣金结算

**权限范围**：
- 可访问：全量楼盘字典（只读）、个人维护楼盘（读写）、个人客源库、优先匹配推荐、营销工具、交易流程工具
- 不可访问：其他经纪人的私有客源库、买卖双方的非公开个人信息

---

### 2.4 中介门店

**角色定义**：在 Fori 平台入驻的线下房产中介机构，管理旗下经纪人团队，作为集体业务单元参与平台生态。

**核心画像**：
- 规模：小型门店（1～5 人）到中型机构（50 人以内）
- 管理诉求：统一管理团队成员、集中查看业绩、控制合规风险
- 核心诉求：平台带来增量客源，降低运营成本，提升团队整体成交率

**核心需求**：
1. 门店主账号统一管理旗下经纪人账号
2. 聚合查看团队所有楼盘维护进展和客源情况
3. 门店维护楼盘享有团队优先匹配权益
4. 门店整体信用评级，对外建立机构品牌
5. 业绩报表、佣金汇总、合规监控

**使用场景**：
- 场景 A：门店管理员新增经纪人账号 → 设定权限 → 经纪人在门店旗下执业
- 场景 B：门店维护某片区全量楼盘 → 团队共享维护权益 → 优先获取该片区客源
- 场景 C：门店管理员查看月度业绩报表 → 分析成交情况 → 调整团队运营策略

**权限范围**：
- 可访问：门店账号管理、团队楼盘维护汇总、团队客源池（门店内共享）、门店业绩报表、佣金汇总
- 不可访问：其他门店的私有客源库与经营数据

---

### 2.5 公证合规第三方机构

**角色定义**：由国家认可的公证机构、法律机构或官方合规服务机构，通过 API 或系统集成方式接入 Fori 平台，参与房产交易的核验、公证、存证环节。

**核心画像**：
- 机构类型：国家认证的公证处、房地产交易监管机构、律师事务所（特定合规资质）
- 工作方式：通过 Fori 系统接口接收核验任务，完成后回传公证结果和数字存证
- 核心诉求：流程标准化、接口清晰稳定、数据安全合规

**核心需求**：
1. 标准化 API 接口，接收来自 Fori 的核验请求
2. 完整的核验任务信息（房源资料、当事人信息、合同文本）
3. 回传公证结果、存证哈希，供平台和用户查询
4. 纠纷仲裁时可通过 Fori 系统调取完整存证记录

**使用场景**：
- 场景 A：Fori 发起房源核验请求 → 机构线上核验资料 → 回传核验结论 → 平台生成合规背书
- 场景 B：交易完成 → Fori 发起存证请求 → 机构完成合同公证存证 → 生成存证编号 → 永久归档
- 场景 C：纠纷发生 → 当事人向公证机构申请调取存证 → 机构通过 Fori 系统导出完整记录

## 3. 核心模块功能需求

---

### 3.0 统一功能点 I/O 契约

以下契约用于统一后续架构设计、接口设计、测试用例和 Agent 任务验收口径。原章节中的流程、字段、权限、状态机继续有效；本节把评审要求的核心功能点归一为一致格式。

### 楼盘字典维护/共享
- 输入：城市/片区/小区/楼栋/单套住宅字段，经纪人或门店认证主体 ID，当前版本号，变更字段，核验材料，纠错申请，合作机构 API 查询参数。
- 处理：校验主体权限和字段级权限；执行重复数据检测、格式校验、逻辑校验、官方数据比对、AI 辅助核验；对并发编辑执行乐观锁和差异合并；生成版本快照；按角色脱敏共享；对合作机构执行协议校验、限流和审计。
- 输出：新增或更新后的楼盘字典记录、版本号、核验状态、维护积分、首建/维护标签、共享查询结果、API 调用审计记录。
- 异常输出：权限不足、版本冲突、字段缺失、重复楼盘、官方数据不一致、核验材料不合规、合作机构协议失效、API 超限、需人工复核的任务单。

### 房源客源甄别流存与匹配
- 输入：房源信息、产权证明材料、卖方身份信息、买方购房需求、联系方式、行为数据、目标区域/楼盘、经纪人维护贡献数据。
- 处理：执行房源真实性、发布者资质、挂牌价合理性、唯一性、活跃度、买家需求有效性和意向强度评分；生成脱敏归档；运行匹配算法；按 P1/P2/P3 规则推送给维护经纪人或团队。
- 输出：房源核验状态、客源质量等级、匹配得分、推送队列、前台展示状态、脱敏历史档案、用户通知。
- 异常输出：未实名、产权不一致、预算不合理、重复客源、备案系统超时、房源过期、经纪人拒绝或超时未响应、进入人工复核或通用匹配池。

### 全主体认证
- 输入：房源资料、房产证/委托书数字副本、买方/卖方身份证和活体识别结果、银行卡绑定信息、购房资格材料、经纪人从业资格证、门店营业执照、房地产经纪机构备案证、法定代表人实名信息。
- 处理：调用身份认证、企业信息、从业资质、官方房产备案、城市购房资格接口；校验共有产权人或委托代理链条；生成主体信用档案和认证等级；记录认证有效期、材料使用范围和审计日志。

### 模块一：全国全层级楼盘字典共建共享体系

#### 功能概述

本模块是平台的**核心数据底座**，目标是建立全国覆盖、标准统一、多方共建、全员透明的房产数据资产库。所有其他模块的功能均依托本模块的数据驱动。

---

#### 1.1 数据层级架构

**五级精细化数据层次**：

```
国家 → 省份 → 城市 → 区县 → 片区 → 小区 → 楼栋 → 单套住宅
```

实际业务中聚焦五层核心数据层：**城市 → 片区 → 小区 → 楼栋 → 单套住宅**

**各层级数据模型**：

**L1：城市层**
- 字段：城市名称、城市编码（行政区划代码）、所属省份、经纬度中心点、城市能级（一线/新一线/二线/三线/四线/五线）
- 维护主体：平台官方
- 更新频率：按国家行政区划变更同步更新

**L2：片区层**
- 字段：片区名称、所属城市、片区类型（住宅区/商住混合/城中村/开发区）、片区边界坐标（GeoJSON）、主要道路边界、片区规划情况、配套设施标签（地铁线路、重点学区、商业综合体、医院等级）
- 维护主体：平台官方 + 经认证经纪人协同维护
- 更新频率：实时协同更新

**L3：小区层**
- 字段：小区名称、所属片区、开发商名称、物业公司名称、建成年份、总楼栋数、总户数、容积率、绿化率、车位比、小区门牌地址、主出入口坐标、配套设施（游泳池/健身房/幼儿园等）、物业费区间、物业评分（平台累积）、小区标签（高端改善/刚需普通/老旧小区/别墅等）、历史交易套数
- 维护主体：经纪人（读写权限）+ 平台核验
- 更新频率：经纪人提交后 24 小时内完成核验，发布生效

**L4：楼栋层**
- 字段：楼栋编号、所属小区、楼层总数、单层户数、建筑类型（高层/多层/别墅/公寓）、电梯数量、楼栋朝向（主朝向）、首层业态（商业/架空层/住宅）
- 维护主体：经纪人（读写权限）
- 更新频率：实时提交，异步核验

**L5：单套住宅层**
- 字段：房号、所属楼栋、建筑面积、套内面积、楼层、朝向、户型（室/厅/卫数量）、户型图（系统标准化版本）、产权年限、当前状态（在售/已售/出租/空置/自住）、当前挂牌价（若在售）、历史成交记录（含成交时间、成交价、成交面积）、最近更新时间、维护经纪人 ID 列表
- 维护主体：经纪人（提交）+ 系统（从官方过户记录同步）
- 更新频率：实时提交；官方数据每日同步一次

#### 1.2 共建维护规则

**1.2.1 准入机制**

| 主体类型 | 准入条件 | 可操作范围 |
|---------|---------|-----------|
| 平台官方 | — | 全部层级读写 |
| 认证经纪人 | 通过个人资质认证 | L3-L5 层读写 |
| 认证中介门店 | 通过机构资质认证 | L2-L5 层读写 |
| 普通注册用户 | 完成实名认证 | L3-L5 层只读 + 提交纠错建议 |
| 未登录用户 | — | L3 层基础字段只读 |

**1.2.2 新增规则**

- 新增 L3 小区：经纪人提交 → 系统自动比对已有数据（防止重复）→ 无重复则进入人工/AI 核验队列 → 核验通过后发布 → 版本 v1.0 建档
- 新增 L4 楼栋：关联已有小区 → 填写楼栋基本信息 → 系统自动核验逻辑合理性 → 发布
- 新增 L5 单套：关联已有楼栋 → 填写户型信息 → 可上传房产证局部截图（脱敏后用于核验）→ 发布

**1.2.3 多人协同编辑机制**

- 同一楼盘 / 单套房源**同时最多支持 5 名经纪人**协同维护
- 采用**乐观锁 + 版本号机制**：每次提交携带当前版本号，若版本号不匹配则提示冲突，展示差异对比界面，由提交方选择合并策略（保留我的修改 / 采用对方修改 / 手动合并）
- 每次成功提交生成新版本快照，保留**完整版本历史**，支持回滚到任意历史版本（需经纪人本人或平台管理员操作）
- 同一字段的最近一次更新者显示为"当前维护人"

**1.2.4 权责匹配与维护激励**

- **首次建档奖励**：率先为某小区 / 楼栋 / 单套建立字典档案的经纪人，获得该数据的"首建者"标签，并获得积分奖励
- **持续维护奖励**：每次提交有效信息更新，经平台核验通过后，获得维护积分
- **维护优先权**：对某楼盘贡献数据最多的 Top 3 经纪人，自动获得该楼盘的**优先匹配权**（详见模块二）
- **惩罚机制**：提交虚假信息一经核实，扣除维护积分、降低维护优先级，情节严重者暂停楼盘字典写权限

**1.2.5 数据核验机制**

- **自动核验**：系统自动检查必填字段完整性、数据格式合法性、逻辑合理性（如楼层不超过楼栋总层数）、重复性（防止同一房源重复录入）
- **AI 辅助核验**：楼盘字典 Agent 对提交内容进行语义分析，识别明显错误（如面积异常、户型描述矛盾）并提示修正
- **官方数据比对**：系统每日从官方房地产交易平台、住建部公开数据同步，自动比对并标记与官方数据不符的字段，触发人工复核流程
- **用户纠错投诉**：任何实名用户可对楼盘数据提交纠错申请，经核实后正确方获得纠错积分

---

#### 1.3 数据共享规则

**共享原则**：
- 平台合规入驻的所有主体均可全量查阅楼盘字典数据（L3-L5 层的基础信息字段）
- 以下字段仅向认证经纪人和认证门店开放：历史成交价格明细、业主联系方式（脱敏）、经纪人维护记录
- 原始上传材料（如房产证截图）不对外共享，仅供平台内部核验使用，完成核验后自动删除原始件

**API 开放规则**：
- 平台提供楼盘数据查询 API，供经认证的合作机构接入（如银行、律所、公证处）
- API 返回脱敏字段，不含个人信息
- API 调用需签署数据使用协议，并受限流保护（每机构每日最高 10,000 次请求）

### 模块二：房源客源甄别流存与智能精准匹配体系

#### 功能概述

基于楼盘字典数据底座，构建平台供需资源沉淀与撮合体系。核心目标：过滤无效供需信息、永久沉淀优质资源、精准匹配撮合、激励经纪人深耕。

---

#### 2.1 多主体信息智能甄别

**2.1.1 房源信息甄别**

**输入**：
- 经纪人或房东提交的房源信息（房屋基本信息、产权证明材料、挂牌价格、联系方式）

**甄别维度与规则**：

| 甄别维度 | 甄别方法 | 未通过处理 |
|---------|---------|----------|
| 产权真实性 | 核验房产证编号是否在官方备案系统中存在 | 标记为"待核验"，暂不上架 |
| 挂牌价合理性 | 对比同楼盘同层级近 6 个月成交均价 ±40% 区间 | 标记"价格异常"，提示房东 / 经纪人确认 |
| 房源唯一性 | 对比同城同小区同楼栋同房号，防止同一房源被重复发布 | 合并为同一房源，通知相关经纪人 |
| 发布者资质 | 核验发布者是否完成实名认证 | 要求先完成认证 |
| 信息完整性 | 必填字段完整率（建筑面积、户型、楼层、挂牌价）≥ 100% | 提示补全信息 |
| 活跃度核验 | 房源发布后 30 天内无更新且无回复视为"疑似过期" | 系统自动发送提醒，3 日内未响应则下架至待重激活状态 |

**输出**：
- 核验通过：房源状态设为"已核验在售"，进入匹配池
- 核验待确认：房源状态设为"待核验"，经纪人/房东收到通知，需补充材料
- 核验未通过：房源状态设为"不合规"，经纪人收到详细说明，可申诉

**异常处理**：
- 官方备案系统查询超时：房源标记"备案核验中"，48 小时内完成异步核验，期间房源可展示但标注"核验进行中"
- 同一房源被同时提交多次：系统自动去重，以最早提交时间为准，后续提交者收到通知并关联到已有房源

---

**2.1.2 房主（卖方）信息甄别**

**输入**：
- 卖方身份证信息、手机号、房产证（产权人姓名需与身份证一致）

**甄别规则**：
1. 实名认证：身份证号 + 活体识别验证（第三方认证服务）
2. 产权核验：核对身份证姓名与房产证产权人姓名一致性（共有产权需所有产权人均提交认证）
3. 黑名单比对：核查是否在平台历史不良行为记录列表中
4. 联系方式有效性：发送验证码，验证手机号真实可用

**输出**：
- 核验通过：卖方获得"实名认证卖家"标签
- 产权不一致：系统拒绝挂牌，提示卖方完善产权材料或联系客服处理共有产权情况

---

**2.1.3 买家（客源）信息甄别**

**输入**：
- 买家提交的购房需求（区域、户型、预算、购房时间线、联系方式）

**甄别维度**：

| 甄别维度 | 规则 |
|---------|------|
| 实名认证 | 实名 + 手机号验证，未认证用户只能浏览，不能进入精准匹配池 |
| 需求有效性 | 预算范围与目标区域房价中位数的合理性校验（预算不低于目标区域在售均价的 50%）|
| 购买意向强度 | 根据用户行为（搜索次数、收藏数量、联系经纪人次数）动态评分 |
| 活跃度核验 | 客源录入后 60 天内无任何互动则降为"待激活"状态，推送重激活提醒 |
| 重复性过滤 | 同一手机号不允许同时存在多条有效购房需求，需先关闭旧需求 |

**输出**：
- 高质量客源：实名认证 + 高意向评分 + 需求合理 → 进入精准客源匹配池，享有优先匹配权
- 普通客源：实名认证但互动行为较少 → 进入普通客源池
- 无效客源：未实名或需求不合理 → 不进入匹配池，提示用户完善信息

---

#### 2.2 合规信息永久流存

**核心原则**：核验通过的房源信息和客源信息，即使交易完成或用户主动撤回，业务历史记录按合规用途留存；其中法定存证类数据不可删除，非存证个人信息按用户请求删除或匿名化，脱敏统计数据可长期用于价格评估和风控。

**房源留存规则**：
- 在售期间：完整信息留存，供匹配使用
- 交易完成：状态变更为"已成交"，成交价、成交时间等关键字段进入楼盘历史成交记录
- 用户撤回 / 下架：前台不可见，后台保留完整历史档案，供核验和纠纷处理使用

**客源留存规则**：
- 有效期内：完整信息参与匹配
- 成交后：客源状态标记为"已成交"，联系方式脱敏归档
- 超期未激活：前台降权，但历史交互记录永久保留

**脱敏规则**：
- 手机号：保留前 3 位和后 4 位，中间替换为 `****`（例：138****5678）
- 身份证号：仅保留年龄信息，其余完全屏蔽
- 家庭住址：仅保留区县级别信息

**留存的业务价值**：
- 历史成交记录驱动在地分层价格评估（模块五）
- 历史客源数据用于分析片区需求热度
- 历史房源数据用于楼盘字典字段补全和核验

**个人信息删除与留存边界**：

| 数据类别 | 处理规则 | 用户权利 | 告知与同意机制 |
|---------|----------|----------|----------------|
| 公证存证数据：合同哈希、时间戳、公证证书、资金监管流水、交易关键节点、纠纷材料包 | 属于交易证据链和法定存证，不可删除，不纳入常规清理 | 用户可查询、下载、申请更正展示错误；不得要求删除已生效存证 | 进入交易流程前单独展示《交易存证告知书》，用户明示同意后方可提交公证 |
| 核验必需原始件：身份证照片、房产证照片、活体识别材料、银行卡核验材料 | 仅用于核验，核验完成后 30 天内删除原始件，仅保留核验结论、证件类型、核验时间和脱敏摘要 | 用户可撤回授权；撤回后停止继续核验，已完成交易的存证结果保留 | 上传前逐项告知用途、保存期限、接收方和撤回方式 |
| 非存证业务记录：未成交房源、已撤回客源、推广素材草稿、浏览/收藏/咨询行为 | 用户申请删除后删除可识别个人信息；为风控、反作弊、争议预防需留存的记录仅保留脱敏摘要 | 用户可删除、撤回授权、关闭推荐画像 | 账号隐私中心提供删除入口，删除前提示会影响匹配、推荐和历史追踪 |
| 统计分析数据：片区热度、价格样本、成交趋势、素材效果聚合指标 | 仅可匿名化留存，不保留可识别个人身份的联系方式、证件号、精确住址 | 用户删除个人信息后，聚合统计不再可回溯到个人 | 隐私政策中说明匿名化后的统计数据用于价格评估、风控和产品改进 |
| 审计日志：涉及个人信息访问、资金操作、授权变更、删除请求处理记录 | 按等保、审计和争议处理要求保存，日志中的个人标识最小化并加密 | 用户可查询删除请求处理结果；审计日志不对外展示 | 隐私政策和操作弹窗告知敏感操作会生成不可篡改审计记录 |

**删除请求处理 SLA**：普通个人信息删除请求 15 个工作日内完成；涉及交易、争议、公证或监管要求的请求，在 3 个工作日内反馈不可删除范围、法律/合规依据和可匿名化处理范围。

#### 2.3 定向优先匹配推荐

**核心逻辑**：楼盘字典中，对某楼盘数据贡献最多的 Top 3 经纪人（或维护团队）获得该楼盘的**优先客源推送权**。

**匹配算法（详细）**：

```
匹配得分 = 
  位置权重（40%）× 区域匹配度
  + 户型权重（20%）× 户型匹配度
  + 预算权重（25%）× 价格匹配度
  + 时间权重（10%）× 购买紧迫度
  + 层级权重（5%）× 品质层级偏好匹配度
```

**推送优先级规则**：

1. **P1 优先推送**：客源的目标楼盘 = 经纪人维护楼盘 AND 经纪人为该楼盘 Top 3 维护者
2. **P2 推送**：客源目标区域与经纪人维护片区重叠，非 Top 3 维护者
3. **P3 通用推送**：全量符合条件的认证经纪人

**推送方式**：
- PWA Web Push 推送通知（标注客源意向强度、匹配度分数）
- 经纪人工作台客源池实时更新
- 每个客源在 P1 级经纪人未响应 4 小时后自动下降至 P2 级，24 小时后至 P3 级

**匹配规则异常处理**：
- 若某楼盘无认证经纪人维护：客源进入全城通用匹配池
- 若客源需求范围跨片区：按各片区权重分配给对应经纪人
- 若经纪人主动拒绝某客源：系统记录，推进至下一级经纪人，不重复推送给同一经纪人

---

#### 2.4 业务正向闭环

```
经纪人维护楼盘数据
        ↓
获得优先客源匹配权益
        ↓
更多客源 → 更多成交
        ↓
成交记录更新楼盘历史数据
        ↓
楼盘数据更完整 → 匹配更精准
        ↓
（循环正向增强）
```

### 模块三：全链路信用认证与第三方公证合规交易体系

#### 功能概述

建立全平台信用体系，从身份认证到交易完成，全链路可信、可核验、可追溯，依托国家级官方体系实现合规极简，通过第三方公证机构完成存证，为争议提供永久有效的法律依据。

---

#### 3.1 全主体闭环信用认证

**3.1.1 房源认证**

**流程**：
1. 经纪人 / 房东提交房源基本信息 + 房产证照片（系统自动 OCR 识别关键字段）
2. 系统自动核验：房产证编号在官方系统备案状态、产权人姓名一致性
3. 地址匹配核验：与楼盘字典地址库进行地址标准化比对
4. 公证机构前置核验（详见 3.2.1）
5. 核验通过后，房源获得"实名认证房源"标签，展示核验机构名称和核验时间

**认证有效期**：认证后 180 天内有效，超期需重新提交更新的房产证明材料

**特殊情形**：
- 共有产权（如夫妻共同持有）：需所有产权人均完成身份认证，房源方可挂牌
- 委托代理出售：需提供公证处出具的委托书，系统留存委托书数字副本

---

**3.1.2 交易双方认证**

**卖方认证**：
- 实名认证（身份证 + 活体识别）
- 银行卡绑定（用于收款验证）
- 产权核验（已在 3.1.1 完成）
- 征信自查授权（非强制，但影响信用评分）

**买方认证**：
- 实名认证（身份证 + 活体识别）
- 银行卡绑定（用于资金监管账户）
- 购房资格核验（按城市限购政策，系统自动调取官方限购查询接口核验社保 / 纳税记录）

**购房资格核验逻辑**：
- 输入：买方身份证号、目标城市
- 处理：调用目标城市住建委 / 公积金管理中心 API（或引导用户自主查询并上传截图）
- 输出：购房资格认证结果（可购 / 不可购 / 需补充材料）
- 异常：API 不可用时，系统提示用户自行查询并上传官方截图，由人工核验

---

**3.1.3 从业者认证（经纪人 + 门店）**

**个人经纪人认证**：

| 认证层级 | 要求 | 对应权益 |
|---------|------|---------|
| L1 实名认证 | 身份证实名 + 手机验证 | 可发布房源、接收客源通知 |
| L2 执业资质认证 | 上传房产经纪人从业资格证或房地产经纪人资格证 | 可参与楼盘字典维护、客源优先匹配 |
| L3 高级认证 | L2 基础上 + 连续 6 个月良好信用记录（无投诉、无失信）| 认证徽章展示、更高匹配优先级 |

**信用评分计算模型**：
```
信用评分（满分 100）= 
  基础分 60（完成 L2 认证后获得）
  + 成交评价分（最高 20分）：客户好评率 × 20
  + 活跃度分（最高 10分）：近 30 天有效操作次数 / 基准次数 × 10
  + 楼盘维护分（最高 10分）：维护楼盘数量 × 权重
```

**门店认证**：
- 营业执照核验（调用企业信息核验 API）
- 房地产经纪机构备案证核验
- 法定代表人实名认证
- 门店经营地址核验（上传门店照片 + 地理位置定位）

---

#### 3.2 第三方公证存证机制

**3.2.1 前置核验**

**触发时机**：买卖双方确认交易意向，进入正式交易流程前

**流程**：
1. Fori 系统自动打包核验资料包（房源信息、买卖双方认证信息、经纪人信息、意向协议草稿）
2. 加密传输至对接公证机构 API 端点
3. 公证机构在线核验（标准响应时间：工作日 1 个工作日内）
4. 公证机构回传核验结论：
   - `APPROVED`：核验通过，附核验编号
   - `PENDING`：需补充材料，附补充清单
   - `REJECTED`：不合规，附拒绝原因
5. Fori 系统根据结论更新交易状态

**输出**：
- 核验通过：生成"合规交易准入证书"（含公证机构盖章、核验编号、有效期），展示在交易详情页

**异常处理**：
- 公证机构 API 不可达：系统切换至备用公证机构，同时通知平台运维
- 核验超时（超过 2 个工作日未响应）：平台向买卖双方发送通知，提供人工跟进选项

---

**3.2.2 交易存证**

**触发时机**：正式合同签订完成（网签完成后）

**存证内容**：
- 三方服务合同（经纪人与买卖双方）原文哈希
- 购房合同（买卖双方）原文哈希
- 关键时间节点记录（意向时间、核验时间、签约时间、付款时间、过户时间）
- 经纪人信用档案快照（存证时点）
- 资金监管账户操作记录（由银行 API 提供）

**技术实现**：
- 将以上数据生成 SHA-256 哈希值
- 附加时间戳（由可信时间戳服务商提供，符合 RFC 3161 标准）
- 将哈希 + 时间戳提交至公证机构，由公证机构签发存证证书
- 存证证书（含公证机构数字签名）存入 Fori 平台永久存证库

**存证查询**：
- 买卖双方和经纪人可凭交易编号查询存证记录
- 公证机构可通过 API 查询存证详情（用于纠纷处理）

---

**3.2.3 争议兜底**

**触发条件**：任何一方（买家、卖家、经纪人）在交易完成后 24 个月内提交纠纷申请

**处理流程**：
1. 当事人在 PWA 内提交纠纷申请 + 上传相关证明材料
2. Fori 平台自动调取该交易全部存证记录，生成"纠纷支撑材料包"
3. 平台在 24 小时内通知对方当事人和经纪人
4. 若双方同意线上调解：平台提供在线调解界面，经纪人和客服协助调解
5. 若无法调解：平台将"纠纷支撑材料包"提交至原公证机构，由公证机构出具纠纷认定意见书
6. 纠纷认定意见书作为仲裁 / 诉讼的参考证据

**存证永久性保障**：
- 存证数据三重备份（平台主库 + 公证机构 + 冷备份存储）
- 任何存证数据标记为"永久留存"，不纳入常规数据清理策略

#### 3.3 国家级合规极简降本体系

**对接官方体系清单**：

| 官方体系 | 对接方式 | 用途 |
|---------|---------|------|
| 住建委网签系统 | API 集成（或提供标准化操作引导） | 完成商品房买卖合同网签备案 |
| 不动产登记中心 | 提供办理指引 + 预约接口 | 房屋过户登记 |
| 税务局 | 提供税费预估工具 + 缴税引导 | 增值税、个税、契税计算和缴纳引导 |
| 银行资金监管 | 接入合作银行资金监管 API | 交易资金安全托管 |
| 公积金管理中心 | 查询接口 + 提取 / 贷款引导 | 公积金购房查询和引导 |

**极简降本措施**：
- 平台内置税费计算器（输入：城市、购房类型、持有年限、购房套数 → 输出：买方税费明细、卖方税费明细）
- 一键生成官方办理清单（按城市差异，列出过户所需材料清单和办理窗口信息）
- 交易进度实时追踪（网签→资金监管→税务缴纳→过户→新房产证）

### 模块四：自媒体智能房源推广营销体系

### 模块五：独创在地分层动态房价评估体系

#### 功能概述

颠覆传统「一二线城市宏观分级定价」模式，基于真实居住价值搭建科学、透明、可拆解、可动态更新的房价测算体系，为买卖双方和经纪人提供统一客观的价格锚点。

---

#### 5.1 核心定价底层理论

**理论命题**：
> 房价走势不存在统一的城市级分化逻辑，核心本质是「微观区域在地分层分化」。同一城市、同一片区、同一街道内，住宅因居住品质差异会形成独立的价格层级与涨跌走势。无论城市能级高低，每个片区都会自然分化出改善圈层、刚需圈层、老旧圈层、资源稀缺圈层，各层级价格涨跌、流动性完全独立，甚至呈反向走势。

**理论对产品设计的指导意义**：
1. 不以城市能级作为定价基准，以**微观片区 × 品质层级**双维度定价
2. 同一小区不同楼栋可能属于不同层级（如小区内高楼层、有电梯、朝南的楼栋 vs 多层无电梯楼栋）
3. 价格走势图必须以**片区层级**为单位展示，而非城市均价

---

#### 5.2 片区品质分层体系

**分层主体**：楼盘字典维护经纪人（人工评级） + 在地分层价格评估 Agent（智能评级） → 综合输出

**四级分层标准**（A/B/C/D 级）：

| 等级 | 定义 | 典型特征 |
|-----|------|---------|
| A 级（改善圈层）| 该片区居住品质最优 | 2000 年后建成、高物业品质、低容积率（≤2.0）、高绿化率（≥30%）、充足车位（≥1:1）、优质学区、地铁近（≤500m）、品牌物业 |
| B 级（中端圈层）| 该片区居住品质良好 | 1995-2010 年建成、物业一般、容积率 2.0-3.0、绿化率 20-30%、车位 0.5:1-1:1、有学区但非重点、地铁 500m-1km |
| C 级（刚需圈层）| 该片区居住品质普通 | 1985-2000 年建成、物业较差、高容积率（>3.0）、低绿化率（<20%）、车位紧张（<0.5:1）、普通学区 |
| D 级（老旧圈层）| 该片区居住品质偏低 | 1985 年前建成或城中村、无物业 / 自管、基本设施老化、无电梯多层为主 |

**分层指标权重**（用于 AI 评级算法）：

| 指标 | 权重 |
|-----|------|
| 楼龄 | 15% |
| 物业品质评分 | 15% |
| 容积率 | 10% |
| 绿化率 | 8% |
| 车位比 | 8% |
| 户型设计合理性 | 8% |
| 采光 / 噪音条件 | 7% |
| 学区等级 | 10% |
| 地铁距离 | 8% |
| 商业配套完善度 | 6% |
| 片区规划利好 | 5% |

**分层更新机制**：
- 每季度由价格评估 Agent 重新计算全库小区的分层等级
- 单次经纪人维护操作（上调 / 下调分层建议）经平台核验后可即时更新
- 重大区域政策变化（如新建地铁、学区调整）触发片区批量重评

---

#### 5.3 动态可视化价格测算逻辑

**5.3.1 基准价格测算**

**输入**：
- 目标房源（或目标楼盘）
- 测算参考时间范围（默认：近 6 个月成交数据）

**处理逻辑**：
1. 定位房源所属微观片区和品质层级（A/B/C/D）
2. 从平台历史成交库中提取该**片区 × 层级**最近 N 笔成交记录（N 默认 30，不足时自动扩展时间范围）
3. 对历史成交单价进行统计分析：
   - 中位数成交单价（作为基准价）
   - 25% 分位和 75% 分位（展示价格区间）
   - 近 3 个月与近 6 个月单价趋势对比（判断涨跌方向）
4. 结合当前市场热度指标（该层级近 30 天挂牌量变化、成交速度）进行动态修正

**5.3.2 个体变量修正**

在基准价基础上，叠加个体变量修正系数：

| 变量 | 修正规则 | 示例 |
|-----|---------|------|
| 楼层 | 低楼层（1-3F）-5% 至 -2%；高楼层（顶楼 -1，高楼层 +2% 至 +5%）| 15F 总层 18F → +3% |
| 朝向 | 纯南向 +3%，东西向 0%，纯北向 -3% | 南北通透 +4% |
| 装修 | 毛坯 0%，简装 +2%，精装 +8%，豪装 +15% | 精装 → +8% |
| 产权 | 70 年产权 0%，40 年产权 -10%，已满 2 年 +1%，满 5 年 +2% | 已满 5 年 → +2% |
| 税费结构 | 高额税费（需买家承担）-3% | — |
| 房源稀缺度 | 该层级该片区在售量 <5 套 → +2%；>50 套 → -2% | — |
| 特殊资源 | 直对湖景/公园 +5%，特殊噪音源 -3% | — |

**5.3.3 价格测算输出（可视化图谱）**

输出格式为可交互的可视化图谱，包含以下组件：

1. **价格区间展示**：雷达图形式，展示基准价、低区间价、高区间价
2. **价格因素拆解图**：瀑布图形式，展示从基准价到测算价的每个变量修正量
   ```
   基准价: 35,000 元/㎡
   + 楼层修正: +1,050 元/㎡（+3%）
   + 朝向修正: +700 元/㎡（+2%）
   + 装修修正: +2,800 元/㎡（+8%）
   - 税费修正: -1,050 元/㎡（-3%）
   = 测算价: 38,500 元/㎡
   ```
3. **历史走势图**：折线图，该片区×层级近 24 个月单价走势（含分位线）
4. **市场热度仪表盘**：挂牌量、成交速度、近期价格变化方向
5. **同层级对标房源**：展示近期 3-5 套成交房源作为参考（脱敏处理）

---

#### 5.4 三方价格赋能

**买家视角**：
- 看房时可在 PWA 内查看该房源的完整价格测算图谱
- 快速判断挂牌价是否在合理区间（"估价偏高 8%"/"估价偏低 3%"/"估价合理"）

**卖家视角**：

**经纪人视角**：

### 模块六：Agent原生智能化技术底座体系

#### 功能概述

平台全程采用 Agent 原生架构开发，基于 OpenClaw 主框架 + Hermes 备选框架，实现三层解耦设计，支持高并发、高稳定、版本同步与框架无缝切换，所有业务独立拆分为专属智能体。

---

#### 6.1 技术框架选型

**主框架：OpenClaw（openclaw.ai）**

选型理由：
- 适配高并发任务调度（如同时处理海量房源核验、视频生成请求）
- 支持多智能体协同（楼盘字典 Agent 与价格评估 Agent 之间的数据流转）
- 异步任务编排（视频生成、批量数据同步等非实时任务）
- 服务自愈机制（Agent 异常后自动重启，保障 7×24 稳定运行）
- 状态持久化（交易流程状态跨会话持久化，不因服务重启丢失进度）

**备选兜底框架：Hermes Agent**

切换条件：
- OpenClaw 发布重大 Breaking Change，适配成本超过切换成本时
- OpenClaw 社区活跃度显著下降，长期维护存疑时
- 特定业务场景下 Hermes 的轻量化编排优势更匹配时

---

#### 6.2 三层解耦架构

**架构设计目标**：业务代码与框架完全解耦，框架升级或切换时，业务层零改动。

**三层结构**：

```
┌────────────────────────────────────────────┐
│               业务 Agent 层                 │  ← 六大业务智能体
│  独立封装，互不耦合，依赖平台内核层接口      │
├────────────────────────────────────────────┤
│               平台内核层                    │  ← 公共底座，与框架无关
│  权限、缓存、监控、日志、分布式锁            │
│  合规存证、数据脱敏、事件总线               │
├────────────────────────────────────────────┤
│               框架适配层                    │  ← 封装 OpenClaw 核心能力
│  标准化 Agent 通用接口，隔离框架底层差异     │
│  框架切换时，仅修改本层，上层无感           │
└────────────────────────────────────────────┘
```

**框架适配层（接口规范）**：

```python
# 标准化 Agent 接口（伪代码）
class AgentInterface:
    def execute_task(task: Task) -> TaskResult: ...
    def schedule_async(task: Task, delay_seconds: int) -> TaskHandle: ...
    def get_state(task_id: str) -> TaskState: ...
    def cancel_task(task_id: str) -> bool: ...
    def health_check() -> HealthStatus: ...
```

业务 Agent 只依赖 `AgentInterface`，不直接调用 OpenClaw SDK。

**平台内核层（能力清单）**：

| 能力模块 | 功能说明 |
|---------|---------|
| 权限管理 | RBAC 权限模型，五类角色的字段级权限控制 |
| 分布式缓存 | Redis 集群，支持多级缓存（L1 本地，L2 分布式）|
| 全链路监控 | 基于 OpenTelemetry，接入 Prometheus + Grafana |
| 结构化日志 | JSON 格式日志，含 trace_id 追踪跨服务调用 |
| 分布式锁 | 基于 Redis 的乐观锁 + 悲观锁（幂等保障）|
| 合规存证 | 对接公证机构 API，管理存证生命周期 |
| 数据脱敏 | 统一脱敏规则引擎，确保出口数据合规 |
| 事件总线 | 基于消息队列（Kafka）的领域事件发布订阅 |

---

#### 6.3 核心业务智能体拆分

**Agent 1：楼盘字典共建 Agent**

- 职责：楼盘新增、多人协同编辑、版本管理、数据核验、数据更新同步
- 核心任务类型：
  - `PROPERTY_DICT_CREATE`：新增楼盘数据
  - `PROPERTY_DICT_UPDATE`：更新楼盘数据
  - `PROPERTY_DICT_VALIDATE`：核验数据真实性
  - `PROPERTY_DICT_SYNC`：与官方数据源同步
- 并发模型：每个任务独立 Worker，支持同一楼盘的并发编辑请求队列化处理
- 状态存储：楼盘版本历史存入专用数据库（写多读多，分库分表）

---

**Agent 2：房源客源甄别流存 Agent**

- 职责：信息核验、数据留存、资源标签化、优先匹配分发
- 核心任务类型：
  - `LISTING_VALIDATE`：房源甄别核验
  - `BUYER_VALIDATE`：客源甄别核验
  - `MATCHING_RUN`：执行匹配算法
  - `MATCHING_PUSH`：向经纪人推送匹配结果
- 并发模型：匹配算法支持批量并发执行，每日低峰期运行全量重匹配

---

**Agent 3：信用认证 & 公证协同 Agent**

- 职责：全主体认证、第三方机构对接、合同存证、信用档案管理
- 核心任务类型：
  - `CREDIT_AUTH_PERSON`：个人实名认证
  - `CREDIT_AUTH_AGENCY`：机构资质认证
  - `NOTARY_REQUEST`：发起公证核验请求
  - `NOTARY_STORE`：合同存证归档
  - `CREDIT_SCORE_UPDATE`：信用评分更新
- 外部依赖：公证机构 API、身份认证第三方 API、企业信息核验 API

---

**Agent 4：合规交易 & 分成结算 Agent**

- 职责：对接官方资金监管、交易流程管控、经纪人收益核算、平台分成统计
- 核心任务类型：
  - `TRANSACTION_CREATE`：创建交易
  - `TRANSACTION_STATE_UPDATE`：更新交易状态
  - `FUND_ESCROW_LINK`：对接资金监管账户
  - `COMMISSION_CALC`：佣金计算
  - `SETTLEMENT_EXECUTE`：结算执行
- 幂等性要求：所有资金操作必须携带幂等键（idempotency_key），防止重复结算

---

**Agent 5：自媒体素材生成 Agent**

- 职责：自动生成房源视频、图文、文案，全渠道分发
- 核心任务类型：
  - `MATERIAL_GEN_VIDEO`：生成短视频素材
  - `MATERIAL_GEN_IMAGE`：生成图文素材
  - `MATERIAL_GEN_COPY`：生成推广文案
  - `DISTRIBUTE_PUBLISH`：全渠道分发发布
- 并发模型：素材生成为 CPU / GPU 密集型任务，独立 Worker 池，支持优先级队列
- 超时处理：素材生成超过 10 分钟自动标记失败并通知用户重试

---

**Agent 6：在地分层价格评估 Agent**

- 职责：片区分层维护、动态价格测算、走势分析、可视化图谱生成
- 核心任务类型：
  - `ZONE_TIER_CALC`：片区层级评估计算
  - `PRICE_ESTIMATE`：单套房源价格测算
  - `PRICE_TREND_ANALYSIS`：片区价格走势分析
  - `PRICE_CHART_GEN`：生成可视化图谱数据
- 定时任务：每日 02:00（低峰期）执行全库价格基准数据更新；每季度执行片区分层重评

#### 6.4 版本同步与框架切换机制

**版本同步原则**：
1. OpenClaw 以 Git Submodule 方式引入，不修改其源码（只在适配层包装）
2. 新版本发布后，先在适配层做兼容性包装，测试通过后更新 Submodule 版本锁定
3. 每个 OpenClaw 版本升级需通过全套集成测试（自动化 CI）验证无破坏性影响

**框架切换演练（SOP）**：
1. 在框架适配层新增 `HermesAgentAdapter`（与 `OpenClawAgentAdapter` 并列）
2. 添加环境变量开关 `AGENT_FRAMEWORK=openclaw|hermes`
3. 灰度切换：先切换 1 个低风险 Agent（如价格评估 Agent），观察 7 天无异常
4. 逐步扩展至全量 Agent
5. 原框架适配层代码保留 90 天后下线

## 5. 商业模式与四方共赢分配机制

### 5.1 核心商业逻辑

Fori 的商业模式基于「薄利多销、生态裂变」原则：

> 以极低的服务佣金抽成换取经纪人口碑和买卖双方信赖，驱动用户规模快速增长，规模效应下获得稳定、持续的平台收入。

**商业模式图**：
```
极低抽成 → 经纪人大量入驻 → 楼盘数据快速完善
    ↓
优质数据 → 精准匹配 → 更多成交
    ↓
更多成交 → 口碑传播 → 买卖双方聚集
    ↓
规模扩大 → 薄利积累 → 平台长期稳定盈利
```

### 5.2 收入来源

| 收入类型 | 来源 | 计费方式 |
|---------|------|---------|
| 交易服务佣金 | 每笔成交交易中抽取 | 按成交总价比例 |
| 高级营销服务费 | 素材生成超额使用 / 付费推广加速 | 按次或按月订阅 |
| 楼盘数据 API 授权费 | 向合作机构开放数据 API | 按调用量计费 |
| 门店认证服务费 | 中介门店入驻认证 | 年费制 |
| 公证机构接入费 | 公证机构接入平台 API | 年费 + 按交易笔数分成 |

### 5.3 四方共赢分配模型（详细）

**以二手房买卖交易为例（成交总价 100 万元）**：

**传统行业模式（对比基准）**：
- 中介总佣金：2%～3%（2 万～3 万元）
- 经纪人实际到手：30%～50%（6,000～15,000 元）
- 平台 / 门店抽成：50%～70%

**Fori 模式**：
- 经纪服务费（买方或卖方支付）：1%～1.5%（1 万～1.5 万元）

**可核算口径**：
- “四方共赢”中的四方固定指：买方、卖方、经纪人、平台。
- 买方和卖方是成本承担与交易收益让渡方；经纪人和平台是经纪服务费分配方。
- 公证存证服务 5% 不是第五个交易分配主体，而是从平台服务费份额中列支或代扣代付的合规服务成本，用于第三方公证机构；对用户报价时仍纳入经纪服务费总额，不另行重复收费。
- 门店参与成交时，经纪人 80% 份额先结算至服务经纪人所属结算主体；门店与经纪人的内部二次分配按双方入驻协议执行，平台不介入门店内部劳动分账，但需记录最终收款主体用于税务和审计。

| 四方主体 | 成本承担 | 收益让渡 / 获得 | 分配比例口径（以 1% 经纪服务费为 100%）| 金额（成交总价 100 万、服务费 1 万）| 核算说明 |
|---------|----------|----------------|--------------------------------------|----------------------------------|----------|
| 买方 | 默认承担 0.5% 服务费，可按交易方案调整为 0%～1% | 获得真实房源、价格图谱、资金监管和交易存证；相对传统 2%～3% 中介费节省成本 | 成本承担方，不参与服务费收入分配 | 默认支付 5,000 元 | 与卖方合计支付服务费 1%，平台禁止经纪人另收隐性费用 |
| 卖方 | 默认承担 0.5% 服务费，可选择卖方全承担 1% | 获得客源、推广、合规交易和更快成交；让渡部分成交收益换取服务 | 成本承担方，不参与服务费收入分配 | 默认支付 5,000 元 | 可在报价策略中选择承担比例，但需在交易确认前明示 |
| 经纪人 | 不承担平台抽成之外的固定平台费；承担自身展业合规责任 | 获得一线服务佣金和维护楼盘带来的优先匹配权益 | 80% | 8,000 元 | 直接服务经纪人或其所属门店结算主体取得，门店内部分账按入驻协议 |
| 平台 | 承担产品、技术、运营、风控、合规审核、公证代付协调成本 | 获得平台服务收入，用于规模化运营和合规基础设施 | 20%，其中平台净服务 15%，公证存证服务 5% | 平台净服务 1,500 元；公证存证 500 元 | 5% 公证存证从平台服务费口径中单列，不向买卖双方重复收费 |

| 分配方 | 分配比例 | 金额（以 1% 即 1 万元为例）| 说明 |
|-------|---------|------------------------|------|
| 一线经纪人 | 80% | 8,000 元 | 直接参与服务的经纪人 |
| 平台净服务 | 15% | 1,500 元 | 技术服务、运营维护 |
| 公证存证服务 | 5% | 500 元 | 从平台服务费口径中单列代付给第三方公证机构 |

**买方**：
- 享受：透明定价图谱、真实房源、低摩擦合规交易、第三方公证保障
- 额外成本：较传统中介节省 1%～1.5% 的中介费（按 100 万房价算：节省 1～1.5 万元）

**卖方**：
- 享受：平台充足客源、智能营销推广、快速成交、合规存证保障
- 额外成本：按选定服务方案支付经纪服务费（可协商买方承担部分）

**经纪人**：
- 享受：比传统门店更高的佣金分配比例（80% vs 传统 30%～50%）
- 激励：维护楼盘数据 → 获得更多优先客源 → 更多成交 → 更多收入

**平台**：
- 收益方式：薄利 × 大规模成交量
- 目标：用低抽成换口碑，实现规模化

### 5.4 定价策略

**经纪服务费标准**：

| 房产类型 | 买方付费比例 | 卖方付费比例 | 说明 |
|---------|------------|------------|------|
| 二手住宅出售 | 0.5%（默认）| 0.5%（默认）| 买卖双方各承担 0.5% |
| 二手住宅出售（卖方全承担）| 0% | 1% | 卖方报价策略选择 |

- 买卖双方均可在 PWA 内举报异常收费，平台在 24 小时内受理调查

## 6. 合规与法律约束

### 6.1 房地产经纪业务合规

- 平台须依法申请《增值电信业务经营许可证》（ICP 证）
- 平台须向当地住建委完成房地产经纪信息平台备案
- 平台入驻经纪人须持有国家认可的房地产经纪从业资格证或房地产经纪人资格证
- 平台入驻门店须持有有效的《房地产经纪机构备案证》
- 平台不得收取定金、保证金等资金托管功能（须依托持牌金融机构资金监管账户）

### 6.2 数据合规

- 用户个人信息收集须明确告知用途并获得明示同意
- 遵守《个人信息保护法》《数据安全法》《网络安全法》
- 敏感信息（房产证、身份证照片）仅用于核验，核验完成后 30 天内删除原始件
- 永久留存的存证数据须符合《电子签名法》要求，确保法律效力

### 6.3 广告与营销合规

- 平台生成的推广文案由 AI 内容审核模块自动过滤《广告法》禁用词汇
- 房源推广内容不得含有：最、第一、全国最低等极限表述
- 经纪人通过平台分发的内容需标注"房产经纪推广"字样（抖音、小红书等平台要求）
- 平台须建立违规内容投诉处理机制，7 个工作日内处理完毕

### 6.4 交易资金监管合规

- 所有交易资金须流经资金监管账户（合作银行的三方监管账户），不得由平台直接持有
- 资金释放须满足合同约定条件，由平台系统验证条件后向银行发送指令（银行实际执行）
- 每笔资金监管操作均须生成银行流水记录，在平台存证

### 6.5 公证存证法律效力

- 平台对接的公证机构须为司法部登记备案的国家正式公证处
- 存证采用区块链存证 + 权威机构时间戳双重保障（非强制，推荐选用方案）
- 存证证书上明确标注：存证内容的哈希值、时间戳、公证机构名称及印章、存证编号
- 纠纷调解未果时，存证材料可直接作为仲裁委员会或法院的参考证据

## 8. 实施决策摘要

### 深度模块设计

- **楼盘字典 Agent**：采用乐观锁 + 版本号机制处理并发编辑，优先多写一致性而非强一致，最终一致即可
- **匹配 Agent**：匹配算法为离线批量 + 实时增量双模式，新客源入库 5 分钟内触发实时增量匹配，全量重匹配每日低峰期执行
- **价格评估 Agent**：分层评级数据为季度级更新，价格测算数据为每日更新，个体测算为实时计算（基于缓存基准 + 实时修正）
- **素材生成 Agent**：GPU 密集型，独立资源池，通过优先级队列防止长尾请求阻塞高优先级任务
- **存证机制**：哈希 + 时间戳 + 公证机构签名三要素缺一不可；存证数据库独立于业务数据库，只追加不修改

### 关键技术决策

- 使用三层解耦架构（框架适配层 / 平台内核层 / 业务 Agent 层），框架切换时业务层零改动
- 分布式锁统一使用 Redis（Redlock 算法），不允许在业务层各自实现
- 所有外部 API（公证、税务、银行）配置熔断器（Circuit Breaker），熔断后降级为人工处理流程
- 个人敏感信息加密字段（AES-256）与非敏感字段分表存储，减少解密操作的性能影响

### 测试策略

- **楼盘字典**：并发写冲突测试（多客户端同时写同一楼盘），版本回滚测试
- **匹配算法**：精准率 / 召回率评估（与手工匹配结果对比），性能测试（10 万客源 × 100 万房源）
- **价格评估**：历史数据回测（用 T-12 个月数据预测 T 月均价，误差 ≤ 15% 为合格）
- **存证流程**：哈希一致性验证，存证证书格式合规性测试，纠纷调取流程端到端测试
- **素材生成**：生成内容合规性自动化扫描（关键词过滤），多分辨率格式兼容性测试
- **全渠道分发**：各平台 API 异常模拟（超时、拒绝、限流），降级处理验证
